\section{Prompt Injection Attacks}
\label{sec:attacks}

Prompt injection attacks exploit the architectural properties described in the
previous section. The core mechanism is that an attacker introduces malicious
instructions into the model's context, causing the model to act on them
instead of the legitimate ones~\parencite{perez2022ignore}. Notably, this does
not require any bug in the software --- the model is functioning as intended,
it is simply following instructions from an unintended
source~\parencite{chen2025struq}. OWASP ranks this as the top security risk
for LLM-integrated applications~\parencite{owasp2024llm}.

\subsection{Direct and Indirect Prompt Injection}

The literature distinguishes two main delivery mechanisms~\parencite{greshake2023indirect}.

\textit{Direct prompt injection} is the more obvious variant: the attacker
interacts with the model directly and attempts to override its instructions
through crafted inputs. Common examples include prompts such as ``ignore
previous instructions'' or attempts to convince the model it is operating in a
different context~\parencite{perez2022ignore}.

\textit{Indirect prompt injection} is generally considered the more dangerous
variant in real deployments. Here, the attacker does not interact with the
model at all --- instead, malicious instructions are placed in content the
model will eventually process as part of its normal operation: a webpage, a
document, an email, or an API response. Since LLM agents and RAG systems
continuously consume such data, users may be entirely unaware that an attack
is occurring~\parencite{greshake2023indirect}. As \textcite{debenedetti2024agentdojo}
demonstrate, untrusted tool outputs are a particularly effective vector in
agentic systems.

\subsection{Why Prompt Injection Works and Its Consequences}

The reason prompt injection works comes down to how LLMs process input. They
operate on token sequences and have no internal representation of where a token
came from. A system instruction and an attacker-injected string are
indistinguishable at the model level~\parencite{liu2023promptinjection}.
This problem is compounded by the fact that modern LLMs are explicitly trained
to follow instructions well --- the same property that makes them useful also
makes them susceptible to manipulation.

What an attacker can achieve depends on the capabilities of the surrounding
system. \textcite{fspib2025} distinguish between \textit{information-level}
and \textit{action-level} impacts. At the information level, attackers can
manipulate outputs, extract sensitive data, or elicit system prompt disclosure.
In agentic systems, a successful injection can additionally trigger tool calls
or take actions on the user's behalf~\parencite{fspib2025}. The attack surface
extends beyond the model itself: injections can propagate into connected
services, functioning as a bridge between natural language manipulation and
more traditional attack vectors~\parencite{pedro2025p2sql}. Real-world
incidents such as the Slack AI attack confirm that these risks are not
theoretical~\parencite{chen2025secalign}.

\subsection{Attack Taxonomy}

Attacks also differ in how the injected content is constructed.

\textit{Heuristic attacks} are handcrafted prompts designed to override the
model's instructions, such as context ignoring or fake completion
techniques~\parencite{jia2025critical}. They are straightforward to construct
and remain effective against many systems.

\textit{Optimisation-based attacks} use gradient-guided search to find token
sequences that maximise the probability of the model complying with the
injected instruction. The Greedy Coordinate Gradient (GCG) method by
\textcite{zou2023universal} is the most widely studied example. These are
considerably harder to defend against and play a central role in evaluating
the robustness of the defences discussed later in this paper.

Before examining those defences, it is necessary to define a threat model that
specifies what attackers are assumed to be capable of and what a secure system
should guarantee.
