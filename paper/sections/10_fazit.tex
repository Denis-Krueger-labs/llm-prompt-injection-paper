\section{Conclusion}
\label{sec:fazit}

The effectiveness of separating instructions and data depends strongly on how
strictly that separation is enforced. Prompt injection is not merely an
implementation flaw, but a structural vulnerability caused by the fact that
modern LLMs process trusted instructions and untrusted data within the same
context window.

Separating instructions and data is therefore an important defence strategy,
but not a complete solution. Fine-tuning approaches such as StruQ and SecAlign
significantly increase the effort required for successful attacks and provide
strong protection against many heuristic prompt injection techniques. However,
these defences remain probabilistic and can still be bypassed by adaptive
attacks designed specifically to target their assumptions. Robustness observed
under standard evaluation conditions therefore does not necessarily translate
into reliable security in realistic adversarial settings.

A central distinction is the difference between semantic and architectural
separation. Semantic approaches attempt to teach the model to recognise and
ignore malicious instructions, whereas architectural approaches enforce trust
boundaries independently of model behaviour. Architectural separation offers
stronger guarantees, but usually requires substantial changes to existing
systems and is harder to integrate into practical deployments.

Prompt injection should therefore be understood as a system-level problem rather
than an isolated weakness of the model itself. As LLMs gain access to tools,
external data sources, and increasingly autonomous workflows, the consequences
of successful manipulation extend beyond incorrect outputs and can affect the
surrounding system as a whole. Increased model capability does not
automatically solve this issue; in some cases, more capable models may also be
better at executing malicious instructions once the security boundary is
bypassed.

Consequently, no single defence currently provides complete protection against
prompt injection. The most realistic strategy is defence in depth, combining
model-level robustness with architectural isolation, least-privilege access
control, monitoring, and human oversight for sensitive operations.

In conclusion, separating instructions and data is a necessary step towards
safer LLM-integrated systems, but it does not eliminate the underlying
vulnerability. Prompt injection remains an open problem, and secure deployment
will depend not only on advances in model capability, but also on stronger
architectural guarantees, more rigorous evaluation methodologies, and careful
attention to emerging threats such as multimodal prompt injection and
increasingly autonomous agents.
