\section{StruQ and SecAlign}
\label{sec:defences}
\label{sec:struq}

Prompt-injection defences operate at different layers. Some modify model
behaviour, while others restrict information flows and actions in the
surrounding application. This chapter examines StruQ and SecAlign as two
model-level approaches to learned instruction--data separation. Both provide
public implementations and published security and utility evaluations
\parencites{chen2025struq}{chen2025secalign}.

\subsection{Motivation for Structured Queries}
\label{subsec:structured-queries}

As discussed in Section~\ref{sec:llms}, an LLM may treat instructional text in
untrusted data as though it were part of the trusted task. A structured query
instead separates the trusted instruction from the data to be processed. In a
document-summarisation application, for example, the instruction defines the
summarisation task while the document forms the untrusted data component.

StruQ preserves this distinction at the application interface and trains the
model to respect it. Its defence therefore combines a secure front end with
specialised instruction tuning
\parencite{chen2025struq}.

\subsection{StruQ}
\label{subsec:struq}

\subsubsection{Secure Front End}
\label{subsubsec:struq-front-end}

The StruQ front end receives the trusted instruction and untrusted data
separately and combines them using reserved delimiter tokens. These include
\texttt{[MARK]}, \texttt{[INST]}, \texttt{[INPT]}, \texttt{[RESP]}, and
\texttt{[COLN]}
\parencite{chen2025struq}.

Because an attacker could attempt to reproduce the delimiters inside the data,
the front end repeatedly removes reserved delimiter strings and
\texttt{\#\#} from the untrusted input before constructing the final query.
This prevents the attacker from directly forging the structural format used by
the trusted front end.

The filter protects the encoding rather than removing all natural-language
instructions. The model must therefore still learn not to follow ordinary
commands that appear inside the data component.

Figure~\ref{fig:struq-architecture} shows how both mechanisms interact. Only
the untrusted data pass through the reserved-string filter before the trusted
instruction and filtered data are combined.

\subsubsection{Structured Instruction Tuning}
\label{subsubsec:structured-instruction-tuning}

Structural formatting alone is insufficient because an ordinary
instruction-tuned model may still follow commands found inside the data.
StruQ therefore uses structured instruction tuning.

The training data contain ordinary structured queries and augmented examples
in which an additional instruction is inserted into the untrusted component.
The expected response continues to follow the trusted instruction rather than
the injected one
\parencite{chen2025struq}.

The augmented examples also include forged instruction sections, delimiter
manipulation, and fake response sections. The front end and model training
therefore serve complementary purposes: the filter protects the reserved
format, while the training reduces compliance with instructional language in
the data.

\subsubsection{Original Evaluation}
\label{subsec:struq-evaluation}

StruQ was evaluated on Llama-7B and Mistral-7B using 208 AlpacaFarm samples.
Most attacks attempted to make the model output a fixed target such as
\enquote{Hacked}
\parencite{chen2025struq}.

For the manually constructed attacks, the defended models achieved Attack
Success Rates (ASRs) between \(0\%\) and \(2\%\). The results were weaker
against an adapted Greedy Coordinate Gradient (GCG) attack, which uses model
gradients during payload construction. StruQ reduced ASR from \(97\%\) to
\(58\%\) on Llama and from \(99\%\) to \(56\%\) on Mistral
\parencites{chen2025struq}{zou2023universal}.

The results show strong resistance to the handcrafted attacks represented in
the evaluation, but incomplete protection against white-box optimisation.
Relative utility changed only slightly in AlpacaEval: from \(67.2\%\) to
\(67.6\%\) on Llama and from \(80.0\%\) to \(78.7\%\) on Mistral
\parencite{chen2025struq}.

StruQ also assumes that the application can identify which input is the trusted
task and which is untrusted data. This is straightforward in structured
workflows, but more difficult in open-ended conversations where users may mix
instructions and data within the same message.

\subsection{SecAlign}
\label{subsec:secalign}

SecAlign pursues the same separation goal but changes the training objective.
The authors argue that supervised structured instruction tuning reinforces the
desired response without explicitly discouraging a response that follows the
injected instruction
\parencite{chen2025secalign}.

\subsubsection{Preference Alignment}
\label{subsubsec:preference-alignment}

SecAlign formulates prompt-injection defence as a preference-optimisation
problem. Each training example contains a prompt-injected input, a preferred
response that follows the trusted task, and a rejected response that follows
the injection.

Direct Preference Optimisation increases the likelihood of the preferred
response relative to the rejected response. The training data are generated by
inserting instructions from other samples into the data component. The
preferred output remains the response to the trusted task, while the rejected
output follows the inserted instruction
\parencite{chen2025secalign}.

Unlike StruQ, SecAlign does not require a newly reserved token vocabulary and
can be applied to existing prompt formats. The application must nevertheless
continue to distinguish the trusted instruction from the untrusted data.
SecAlign changes model behaviour; it does not infer trustworthy provenance on
behalf of the application.

\subsubsection{Evaluation and Comparison with StruQ}
\label{subsec:secalign-evaluation}

SecAlign was compared with StruQ across five Llama and Mistral model variants
using 208 AlpacaFarm security samples. Under the default criterion, an attack
succeeded when the response began with \enquote{Hacked} or
\enquote{hacked}
\parencite{chen2025secalign}.

Across three optimisation-free attacks, SecAlign's maximum ASR was \(0\%\) for
every evaluated model. Across GCG, AdvPrompter, and NeuralExec, the maximum ASR
for each model ranged from \(27\%\) to \(60\%\) under StruQ and from \(1\%\) to
\(14\%\) under SecAlign. SecAlign therefore reduced the maximum
optimisation-based ASR by more than a factor of four for each evaluated model
\parencite{chen2025secalign}.

The authors also evaluated utility with AlpacaEval2 over 805 prompts. The
reported win rates remained close to those of the corresponding undefended
models
\parencite{chen2025secalign}.

These results show a substantial within-study improvement over StruQ.
Nevertheless, the evaluation depends on the selected models, attack suite,
prompt format, target-prefix success criterion, and white-box assumptions.
SecAlign also remains dependent on defensive fine-tuning and correctly
separated application inputs.

\subsection{Comparison and Remaining Limitations}
\label{subsec:struq-secalign-comparison}

StruQ combines reserved delimiter filtering with supervised structured
instruction tuning. SecAlign instead uses preference optimisation to favour
responses to the trusted task over responses that follow an injection.

Both approaches substantially reduce attack success in their original
evaluations. SecAlign performs better against the optimisation-based attacks
included in its study, while both report limited utility degradation under
their respective benchmarks.

Neither approach creates an independently enforced authority boundary. The
decision to follow the trusted instruction remains part of probabilistic model
behaviour, and the reported effectiveness depends on the models, attacks,
tasks, success criteria, and attacker access. Chapter~\ref{sec:limits}
examines these dependencies and later defence-adaptive evaluations.
