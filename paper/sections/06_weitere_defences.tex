\section{Further Defence Mechanisms}
\label{sec:defences}

Beyond StruQ and SecAlign, there are several other defence mechanisms that are
not based on model fine-tuning. These approaches instead rely on enforcing
security at the architectural level. This section discusses three such
approaches in detail.

\subsection{F-secure LLM Systems}

One such system is the f-secure LLM system, a representative framework based on
Information Flow Control (IFC)~\parencite{wu2024systemlevel}. This system
separates the application into three parts. When a prompt is received, it is
first given to the LLM planner, which generates execution steps based on the
query and the previous steps. These steps are then passed to the rule-based
executor, which initiates a new LLM instance to complete the query based on the
generated plan.

During execution, the security monitor oversees the process by checking both
the generated steps from the planner and the output of the LLM instance for
suspicious or incorrect content. It does this by adding labels for untrusted and trusted information, with the goal of restricting the planner to trusted information when generating the steps the actual LLM model has to follow \parencite{wu2024systemlevel}.

\subsection{Agent Security}

In the domain of agentic systems, RTBAS (Robust Tool-Based Agent System) and
CaMeL (Capabilities for Machine Learning) provide specialised protections for
tool-calling scenarios. RTBAS employs dependency screening to identify which
parts of a model's history are relevant for generating a tool call. It uses "LM-as-a-judge" or attention-based screeners to mask irrelevant regions and selectively propagate security metadata, thereby reducing unnecessary tainting of tool calls while maintaining utility \parencite{zhong2025rtbas}.

Similarly, CaMeL extracts control and data flows from queries as pseudo-Python
code and uses a custom interpreter to enforce capability-based policies. By
assigning unforgeable tags, or capabilities, to every value, the system can
block tool execution, such as sending an email, if private data is passed to
unauthorised recipients, regardless of what an injected prompt might command
~\parencite{debenedetti2025defeating}.

\subsection{Additional Conventional Defences}

More traditional ad-hoc defences continue to serve as lightweight layers,
though their effectiveness is generally lower~\parencite{chen2025secalign}.
These include Instruction Defense, Post-Prompting, XML Tagging, and Sandwich
Defenses~\parencite{liu2023promptinjection}. While these can mitigate basic
heuristic attacks, they are frequently circumvented by sophisticated payloads
generated by automated methodologies such as GCG or adaptive optimisation
~\parencite{chen2025secalign}.

Collectively, these approaches illustrate that effective defence requires more
than model-level adjustments. The table makes the core trade-off visible:
fine-tuning methods are easy to deploy but fail under adaptive pressure, while
architectural approaches offer stronger guarantees at the cost of substantially
higher integration complexity. The most effective strategy therefore combines
multiple layers --- input filters, architectural isolation, and capability
tracking --- rather than relying on any single mechanism.
