\section{Threat Model}
\label{sec:threat}

To evaluate defences meaningfully, it is necessary to be precise about the
assumed system and attacker. The effectiveness of any defence depends on what
capabilities an attacker has and what the target system looks like.

\subsection{System Model}

This paper considers LLM-integrated applications where developer instructions
and untrusted external content are processed within the same context window.
This covers a wide range of deployed systems: RAG pipelines, AI assistants,
document summarisation tools, and autonomous agents that interact with external
APIs~\parencite{debenedetti2024agentdojo, ruparel2026survey}.

The defining characteristic of these systems is that the model receives both
its instructions and potentially adversarial data in the same format, with no
built-in mechanism to distinguish between them~\parencite{chen2025struq}. A
malicious instruction embedded in a retrieved document is structurally
indistinguishable from a legitimate developer instruction, which is the
fundamental condition that makes prompt injection possible.

\subsection{Attacker Model}

The attacker does not require access to the model or the application itself.
It is sufficient to control any external source the system will eventually
consume --- a website, an email, an uploaded document, or an API
response~\parencite{greshake2023indirect}. This is a low barrier to entry,
which is one reason indirect injection is considered the more realistic threat
in practice.

Attacker knowledge varies across scenarios. A strong-knowledge attacker has
access to the prompt structure and can craft precisely targeted injections. A
weak-knowledge attacker controls only the injected content and operates without
visibility into the surrounding
context~\parencite{pandya2025astra}. On the technical side, attackers range
from simple handcrafted prompts to fully automated optimisation-based methods
such as GCG~\parencite{zou2023universal}.

\subsection{Security Goals and Assumptions}

A system that defends against prompt injection should ensure three things: that
only trusted instructions influence model behaviour, that sensitive context
data is protected, and that agentic systems do not take unintended
actions~\parencite{chen2025struq, chen2025secalign, debenedetti2024agentdojo}.

This paper is scoped to text-based prompt injection in standard LLM-integrated
deployments. Traditional software vulnerabilities, infrastructure-level
attacks, and multimodal vectors are out of scope. A shared assumption across
most defences is that the system prompt itself remains under developer control.
If that assumption breaks --- if the trusted instruction source is compromised
--- most current mitigations become ineffective.

With this threat model established, the following sections examine how the
research community has approached mitigation, starting with defences that
attempt to enforce a structural separation between instructions and data.
